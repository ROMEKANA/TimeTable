# 作業ルール

## Git運用

- Gitの実操作はユーザーが行う。
- Codexは `git add`、`git commit`、`git push` などのGit書き込み操作を原則実行しない。
- 変更が一区切りついたら、Codexはコミットするタイミングと日本語コミットメッセージ案を提示する。
- コミットメッセージ案は、コピペしやすいようにメッセージ本文だけをコードブロックで出す。
- コミットメッセージは日本語で、変更内容が分かる短い文にする。
- 新しいプロジェクトを作ったら、最初にGitリポジトリとして初期化することを提案する。
- ビルド成果物、一時ファイル、ローカル環境依存ファイルはコミットしない。そのための `.gitignore` も用意する。
- 既存のユーザー変更を勝手に戻さない。
- コミット前に `git status` を確認し、意図したファイルだけを含めるよう案内する。

## 作業姿勢

- まず既存の構成と習慣を確認してから変更する。
- 変更は目的に対して小さく保つ。
- 動作確認できる場合は確認してから完了報告する。
- エラーや制約がある場合は、原因と次に取れる手を短く説明する。
- 自分でできることは、なるべく追加確認を待たずに実行して報告する。
- 覚えておくべき追加の指示は、適宜このファイルにまとめる。

## コード編集

- 読みやすさを優先し、不要な抽象化は増やさない。
- 設定で変えられると便利な値は、コード直書きより設定ファイル化を検討する。
- ユーザーが編集する設定ファイルは、例を分かりやすく保つ。
- コメントは必要な場所にだけ短く書く。
- 文字化けして見えるのは、ユーザーには正常に見えていることがある。削除や変更する前に、文字コードの設定を見直したり、ユーザーに確認したりすること。基本的にUTF-8を使用。
- 勝手な削除は慎む。コードの削除の際は、コメントアウトするようにする。
- もう一度言う、コードの削除はコメントアウトで対応する。

## 検証

- 可能ならビルドやテストを実行する。
- 実行できなかった検証がある場合は、その理由を報告する。
- ゲームModなど外部アプリ上での確認が必要な場合は、確認手順とログ取得方法も添える。

## 2026-07-16時点のバグ候補・完成前に確認したいこと

- 生徒名・講師名の重複チェックがない。予定表、敬称、教材、給与、一覧切替の多くが「学年+生徒名」または「講師名」で検索しているため、同名データがあると別人の情報を拾う可能性がある。安定したIDを持たせるか、少なくとも保存時に重複警告を出したい。
- 生徒/講師一覧の項目切替時の自動保存は、保存後に名前で選択先を探し直している。同名がある場合、意図した行ではなく先に見つかった行を読み込む可能性がある。重複名対策と一緒に見直す。
- `MaxStudentPerTeacher`、曜日、時限を減らす操作は `resize()` で後ろのデータを落とす。現在の確認は「元に戻す履歴が消える」だけなので、実データが消える操作として明示確認を追加したい。
- 講師列の削除も、列内の授業データを消す割に確認が undo 履歴の破棄中心になっている。削除対象の曜日・講師名を出した確認を入れたい。
- 曜日/時限の編集では、編集欄の未反映セル内容に対して `updateCell()` を呼ばずに表を組み替えている。曜日/時限変更前に選択セルの編集内容を反映する必要がある。
- 生徒一覧の保存・削除後は時間割側の生徒コンボの現在値を維持するようにしたが、講師一覧の保存・削除後に更新される時間割側の講師コンボには同じ保護がない。候補から消えた講師名が空白になり、直後の講師列名変更で列名を空にする可能性がある。
- `updateCell()` が印刷、PDF、給与、予定表コピー、生徒/講師一覧からのコピーなど多数の入口で無条件に呼ばれる。時間割編集欄の表示が別タブ操作で古くなったり空白化した時、選択中セルへ意図せず書き戻す可能性が残る。編集欄の dirty 管理か、スケジュールタブ外での更新条件を整理したい。
- マスターデータ変更後の `refreshAfterMasterDataChanged()` は生徒編集欄を `clearStudentEntry()` する。生徒編集中にマスターを編集すると、保存前の生徒編集内容が消える可能性がある。マスター編集前後に生徒/講師の未保存編集を反映する導線が必要。
- 壊れた `.schedule` を読み込むと警告後に新規週として表示される経路がある。直後に保存すると壊れたファイルを上書きし得るため、破損ファイルの退避や「新規として開いている」表示がほしい。
- schedule/master/students/teachers/school/appState の保存は通常の `QFile` 書き込みで、途中失敗するとファイルが壊れる可能性がある。重要データは `QSaveFile` で原子的に保存したい。
- `scheduleMatchesSavedFile()` は JSON 文字列の完全一致で未保存判定している。内容が同じでも整形差や古い保存形式の差で保存確認が出る可能性がある。必要なら構造比較に変える。
- 講師向け日別印刷のヘッダ曜日は、対象週ファイルの `days` ではなく現在開いている時間割の `days` を参照している箇所がある。週ごとに曜日ヘッダが違う場合、表示曜日がずれる可能性がある。
- 「この週を来週にコピー」はコピー先を `今日基準の来週` にしている。過去/未来の任意週を開いていると、表示週の翌週ではなく今日基準の来週へ保存されるため、UI文言と期待のズレが起きやすい。
- 生徒予定表・講師予定表のコピーは現在開いている週の `scheduleEntries()` を使う。今後、対象週選択や複数週出力を増やすなら、対象週ファイルを明示的に読み込む設計へ寄せたい。
